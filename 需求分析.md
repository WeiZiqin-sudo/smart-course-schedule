# 校园学生课程表产品完整实现方案（基于你提供的单节课数据结构）
## 一、先梳理现有字段含义 & 缺失字段确认（核心待确认点）
### 1. 现有字段释义
```json
{
  "kchId":"0801105003", // 课程号（教务系统唯一课程编码）
  "id":null, // 单条课表记录主键id，当前为空待生成
  "name":"材料力学C", // 课程名称
  "teacher":"02467/方建士/教授", // 教师编码/姓名/职称，分隔符/
  "time":"3-4", // 节次：第3、4连堂课
  "place":"南A304", // 上课教室地点
  "week":10, // 周次：仅第10周上课
  "day":"星期六" // 星期几上课
}
```

### 2. 必确认：当前结构缺陷，需要补充哪些字段？
#### （1）周次规则模糊（最大痛点）
现在只有 `week:10`，只能表达**单周上课**，校园课程分多种模式，你要确认业务规则：
1. 是否存在**单双周**上课？如1,3,5奇数周 / 2,4,6偶数周
2. 是否连续多周？1-16周全学期、1-8周、9-16周
3. 存储设计方案二选一：
   - 方案A：week字段改为字符串 `1-10`、`1,3,5,7`、`2-16双` 统一解析
   - 方案B：拆分成 startWeek、endWeek、weekType（全部/单周/双周）三个数字字段
> 待确认：你们教务原始数据是只给单周数字，还是有区间、单双周标识？

#### （2）星期day字段存储优化
当前day是中文「星期一、星期六」，前端排序、筛选麻烦，建议：
- 数据库存数字：1=周一、2=周二…7=周日，中文只做展示
> 待确认：后端是否统一转数字存储，前端再映射中文？

#### （3）节次time字段拓展
`3-4` 代表3、4节连堂，需确认：
1. 单节课格式 `time="2"`、多连堂 `1-2` 是否是唯一格式？
2. 每一节对应固定时间段（如第1节8:00-8:45），是否需要维护**课时时间表配置表**？
> 待确认：是否需要绑定每节课的起止时间，用于展示几点上课、下课提醒？

#### （4）教师字段拆分
`02467/方建士/教授` 拼接字符串，检索、筛选教师极不方便，建议拆分3个独立字段：
- teacherId：02467
- teacherName：方建士
- teacherTitle：教授
> 待确认：后端解析时是否自动按`/`分割存储，还是教务接口会分开返回？

#### （5）缺失核心业务字段（学生端必备）
现有数据完全缺少学生身份关联信息，必须补充：
1. studentId：学生学号（核心，区分每个学生课表）
2. term：学期，如2026-2027-1（区分上下学年，多学期课表共存）
3. credit：学分（学生需要看学分）
4. examType：考核方式（考试/考查）
5. remark：备注（调课、停课、实验课说明）
> 待确认：教务接口是否能返回学分、学期、考核方式？没有的话是否需要手动维护？

#### （6）主键id设计
当前`id:null`：
- 单条课表记录自增id（唯一主键）
- kchId是课程编号，一门课会生成多条周次记录，不能做主键

## 三、后端业务逻辑流程（教务数据同步 + 学生查询）
### 1. 数据来源同步流程
1. 对接学校教务API，拉取每个学生每学期所有课程原始数据（你给的JSON结构）
2. 数据清洗：
   - 拆分teacher字符串为工号/姓名/职称
   - day中文转数字1-7
   - 解析week单周，转换为统一week_rule区间格式
3. 增量入库：按student_id+term+kch_id去重，更新/新增课表记录
4. 同步调课数据覆盖原有记录

### 2. 学生端查询逻辑（核心功能）
1. 根据登录学生学号 + 当前选中学期，查询全部课程
2. 筛选规则：
   - 指定第几周 → 匹配week_rule过滤本周要上的课
   - 按周一到周日、节次从小到大排序
3. 多维度筛选：按课程名、教师、教室搜索课程

2. 周次筛选逻辑：输入当前周，如何匹配单周/双周/区间课程？
3. 数据同步频率：每日自动同步，还是手动一键同步教务数据？
